# Этап 2. Требования к системе

## 1. Функциональные требования (ФТ)

Приоритизация по методу MoSCoW: Must (Обязательно), Should (Желательно), Could (При наличии ресурсов).

| ID | Название | Описание | Приоритет |
| :-- | :-- | :-- | :-- |
| **FR-1** | Просмотр расписания | Система должна отображать список тренировок на ближайшие 7 дней (горизонт планирования по умолчанию). | Must |
| **FR-2** | Фильтрация слотов | Пользователь может фильтровать тренировки по формату (Болдеринг / Трассы с верёвкой). | Should |
| **FR-3** | Групповое бронирование | Пользователь может забронировать до 3-х мест за одну транзакцию (себя + до 2 гостей). | Must |
| **FR-4** | Выбор снаряжения | Для каждого места в брони должен быть выбран тип инвентаря: «Своё» или «Прокат». | Must |
| **FR-5** | Контроль двойных лимитов | Система должна проверять наличие мест по двум ресурсам: 1) Вместимость группы (8/16 чел) и 2) Наличие снаряжения (всего 12 комплектов). | Must |
| **FR-6** | Статусная модель отмены | Система должна определять тип отмены на основе времени: ≥2 ч до старта — «Ранняя» (место освобождается), <2 ч — «Поздняя» (место занято). | Must |
| **FR-7** | Авторизация | Доступ к функциям бронирования осуществляется по номеру телефона через SMS OTP. | Must |
| **FR-8** | Отмена скалодромом | Система должна отображать записи, отмененные администратором (профилактика), со статусом «Отменена скалодромом». | Must |

## 2. Нефункциональные требования (НФТ)

| ID | Категория | Требование |
| :-- | :-- | :-- |
| **NFR-1** | Производительность | Время отклика API при получении списка слотов не должно превышать 2 секунд (p95) при стабильном соединении. |
| **NFR-2** | Надежность | Бэкенд должен выступать «источником истины» (Source of Truth) для всех лимитов и правил отмены. |
| **NFR-3** | Доступность (UX) | Дизайн интерфейса должен соответствовать принципам Mobile-first и обеспечивать контрастность текста не ниже стандарта WCAG AA (для работы при ярком освещении). |
| **NFR-4** | Безопасность | Для управления сессиями должны использоваться JWT-токены. Передача данных только по TLS (HTTPS). |
| **NFR-5** | Целостность | Реализация механизма `Idempotency-Key` для метода создания бронирования для предотвращения дублей при повторных запросах. |
