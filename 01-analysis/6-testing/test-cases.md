## 3. Сценарные тест-кейсы (End-to-End)

### TC-S-01: Групповое бронирование с дефицитом снаряжения
**Предусловия:** Свободных мест: 10, Свободного снаряжения: 1.
1. Выбрать тренировку.
2. Выставить в степпере 2 человека.
3. Для «Место 1» выбрать «Прокат».
4. Для «Место 2» попытаться выбрать «Прокат».
**Ожидаемый результат:** Опция «Прокат» для второго места заблокирована. Система заставляет выбрать «Своё снаряжение» для второго гостя.

### TC-S-02: Идемпотентность при сбое сети
**Предусловия:** Клиент на экране SCR-004, заполнил форму.
1. Нажать кнопку «Записаться».
2. Имитировать обрыв связи (timeout).
3. Нажать кнопку «Повторить» (используется тот же Idempotency-Key).
**Ожидаемый результат:** Система возвращает ответ 201 Created. В базе данных создана ровно ОДНА запись.

### TC-S-03: Граничное условие правила 2 часов (Ранняя отмена)
**Предусловия:** Старт тренировки в 12:00. Текущее время 10:00:00.
1. Нажать «Отменить» на экране SCR-006.
2. Подтвердить в шторке BS-003.
**Ожидаемый результат:** Сервер возвращает статус `cancelled`. Место в группе и 1 комплект снаряжения возвращаются в фонд клуба (ранняя отмена включительно).

## 4. Отчет об обнаружении и исправлении логических багов

В ходе аналитического ревью (Day 3) были выявлены и устранены следующие критические уязвимости:

### 🐞 BUG-01: Ghost Bookings (Удержание ресурсов)
- **Симптом:** При удалении аккаунта (SCR-007) активные брони оставались в базе, блокируя места и тапки навсегда.
- **Исправление:** В метод `DELETE /profile` добавлена триггерная логика каскадной отмены всех активных броней пользователя.
- **Требование:** R-009.

### 🐞 BUG-02: Time Threshold Race (Гонка со временем)
- **Симптом:** Если клиент открыл экран отмены в 09:59 (бесплатно), а нажал кнопку в 10:01 (платно), возникал конфликт ожиданий.
- **Исправление:** Внедрена логика **Server-side Re-validation**. Статус отмены вычисляется строго по времени поступления запроса на сервер (Source of Truth).
- **Требование:** FR-17.

### 🐞 BUG-03: Лимит 4 места (Ошибка Дизайна)
- **Симптом:** В макете SCR-004 была возможность выбрать 4 места.
- **Исправление:** ТЗ (FR-12) приведено в приоритет над дизайном. Лимит в UI и API жестко ограничен 3 местами.

## 5. Использованные промпты (Prompts History)

1. **Для компонентных проверок:** *"Ты — Senior QA. Изучи ТЗ на экран SCR-002. Составь список тест-кейсов для проверки состояний Full (мест нет) и Offline, учитывая физический контекст использования (руки в магнезии, плохой свет)."*
2. **Для Race Condition:** *"Проанализируй Use Case записи (UC-1). Спроектируй тест-кейс на ситуацию, когда 2 пользователя одновременно пытаются забронировать последний комплект снаряжения. Опиши ожидаемое поведение API (код 409)."*
3. **Для баг-репортов:** *"Проведи аудит связи между функцией удаления аккаунта и логикой лимитов снаряжения. Найди потенциальные 'дыры', которые могут привести к потере прибыли клуба."*
