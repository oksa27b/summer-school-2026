# Бизнес-требования

> Этап 2. Зачем бизнесу продукт: цели, метрики успеха, ограничения верхнего уровня.

> **Скоуп — клиентское мобильное приложение и API для него.** Бизнес-цели, обслуживаемые
> существующей инфраструктурой (админка владельца, интерфейс инструктора, управление зонами), 
> в этот список не входят.

> **О нумерации требований.** ID требований (BR-NN, M-NN) сохраняются стабильными для
> трассировки. В нумерации есть пропуски — это следствие осознанного сужения скоупа (часть ранее
> заведённых требований вынесена в развитие/Phase 2 или относится к существующей инфраструктуре).
> Нумерация намеренно не нормализуется, чтобы не ломать связи в таблице-источнике и смежных
> документах.

> **Роли в текущей поставке.** В текущую поставку входит только роль **Клиент** (самостоятельная
> запись на тренировки через мобильное приложение). Интерфейсы инструктора и администратора/владельца
> — **вне скоупа текущей поставки**: они обеспечиваются существующей инфраструктурой. Это
> осознанное решение в рамках MVP-подхода и жёсткого срока: продукт фокусируется на клиентском
> сценарии, который снимает основную боль (хаос записи в Telegram и двойные брони), а серверные
> функции остаются на стороне существующего бэкенда.

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |

## Метрики успеха

> Целевые значения — разумные ориентиры для MVP (точные пороги фиксируются как допущение, при наличии данных уточняются).

| ID | Метрика | Целевой ориентир | Связь с требованиями |
| :-- | :-- | :-- | :-- |
| M-1 | Доля записей, сделанных клиентами онлайн самостоятельно (от общего числа записей). | ≥ 75% к концу первого месяца работы | BR-2 |
| M-2 | Число двойных броней / конфликтов по местам и снаряжению. | 0 (полное устранение) | BR-1 |
| M-4 | Простой мест из-за поздних отмен (доля поздних отмен от общего числа броней). | Снижение относительно текущего уровня (целевой ориентир ≤ 8%) | BR-5 |
| M-6 | Соблюдение сроков запуска. | Готовность системы к началу сезона (через 2 месяца) с критичным набором функций | BR-7 |

## Ограничения верхнего уровня

- **Срок и бюджет.** Жёсткая цель — начало сезона (~2 месяца); бюджет ограничен → строго MVP-подход, дополнительные функции откладываются. (Бриф; Ответы заказчика № 17)
- **Оплата офлайн на старте.** Реализация онлайн-эквайринга в приложении вне границ текущего скоупа. (Ответы заказчика № 9)
- **Mobile-first.** Приложение ориентировано на использование со смартфонов; данные потребляются из существующего бэкенда через API (Black-box). (Ответы заказчика № 12, 16)
- **Вне скоупа.** Интерфейсы для инструкторов/администраторов, функции управления расписанием, автоматическая профилактика зон и программа лояльности — вне текущей разработки. (Описание домена → «Границы скоупа»)
- **Оценки инструкторов — Phase 2 (подтверждено заказчиком).** Система рейтингов и отзывов осознанно вынесена во вторую фазу. Заказчик подтвердил перенос (2026-06-25), чтобы обеспечить своевременный запуск основного функционала записи. (Описание домена → «Границы скоупа»; решение R-001)
