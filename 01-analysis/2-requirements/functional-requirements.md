# Функциональные требования

> Этап 2. Что система должна делать (функции, поведение). 
> Не путать с нефункциональными ([non-functional-requirements.md](non-functional-requirements.md)).

Приоритеты по методу MoSCoW: **Must** — обязательно к запуску; **Should** — желательно при наличии времени; **Could** — приятное дополнение; **Won't** — вне границ скоупа.

> **О нумерации требований.** ID требований (FR-NN) сохраняются стабильными для трассировки. В нумерации есть пропуски — это следствие осознанного сужения скоупа (часть ранее заведённых требований вынесена в развитие/Phase 2 или относится к существующей инфраструктуре). Новые требования продолжают сквозную схему FR-NN без переиспользования занятых номеров.

## Границы скоупа (клиентское приложение + API)

Скоуп — **клиентское мобильное приложение и API для него**. Функции инструктора, владельца и администратора обеспечиваются **существующей инфраструктурой** и здесь не описываются.

Следствия для скоупа:
- **Слоты (тренировки)** приходят из существующего бэкенда через API. Экранов создания и редактирования расписания в клиентском приложении нет.
- **Форматы (Болдеринг/Верёвка) и инструкторы** доступны клиенту как справочные (read-only) данные из API — клиент видит их в карточке слота, но не управляет ими.

## Модель цены (учёт)

> Итоговая стоимость брони рассчитывается сервером по модели:
> `price_total = price × seats_count + rental_price × rental_count`,
> где `price` — цена тренировки, `seats_count` — число мест, `rental_price` — цена проката комплекта, `rental_count` — число прокатных комплектов.
